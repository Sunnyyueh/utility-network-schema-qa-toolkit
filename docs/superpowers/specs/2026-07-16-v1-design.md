# Utility Network Schema QA Toolkit V1.0 Design

## Objective

Deliver a genuinely installable, license-free Python toolkit that validates exported Utility Network schema and migration artifacts without requiring ArcGIS, ArcPy, a geodatabase connection, or a proprietary runtime. V1.0 provides a stable Python API, command-line interface, deterministic findings, and JSON, CSV, Markdown, and HTML reports.

The toolkit is intended for GIS analysts, migration developers, civil and environmental engineering teams, reviewers, and CI workflows. It validates structured exports and workbooks; it does not edit production datasets or execute data loads.

## Product Boundaries

V1.0 includes:

- Python 3.10+ package with a `src/` layout and the `un-schema-qa` console command.
- CSV, JSON, YAML, and XLSX readers.
- Column-alias recognition for common Esri Data Reference and Mapping workbook terminology.
- Canonical typed models for schemas, domains, mappings, filters, asset classifications, engineering context, rules, dirty areas, findings, and run summaries.
- Schema, mapping, filter, domain, asset classification, Data Reference, attribute-rule, network-rule, and dirty-area validators.
- Explainable civil-engineering classification rules supplied through YAML.
- Deterministic JSON, CSV, Markdown, and HTML reports.
- Example projects for water, wastewater, and stormwater.
- User, contributor, architecture, configuration, input-format, and security documentation.
- Automated unit, integration, CLI, and report tests plus GitHub Actions CI.

V1.0 excludes:

- ArcPy or any dependency that requires an ArcGIS license.
- Direct geodatabase, feature service, or enterprise database access.
- Web UI, hosted service, database persistence, or distributed execution.
- Data loading, schema mutation, reconcile/post, or credential handling.
- AI matching, hidden scoring, or execution of arbitrary Python expressions.
- A general-purpose plugin framework or custom DSL beyond the documented YAML rule format.

## Architecture

The package uses a modular validation pipeline:

```text
project.yml + structured input files
              |
              v
readers -> normalization -> canonical ProjectData
                                   |
                                   v
                            validator registry
                                   |
                                   v
                         findings + run summary
                                   |
                                   v
                      JSON / CSV / Markdown / HTML
```

Package boundaries:

- `models`: immutable or validated domain models and enums.
- `readers`: format-specific parsing and source-location capture.
- `normalization`: column aliases, data types, booleans, nulls, paths, and identifiers.
- `filters`: safe SQL-style tokenization and simple predicate analysis; never executes a query.
- `rules`: engineering-condition evaluation using an explicit allowlist of operators.
- `validators`: isolated validation units that consume canonical models and return findings.
- `engine`: validator selection, severity policy, deduplication, sorting, status calculation, and run metadata.
- `reporters`: deterministic serialization and presentation.
- `api`: stable high-level Python functions.
- `cli`: user-facing commands and exit-code handling.

Validators never open files and reporters never re-run validation. This keeps input compatibility, validation behavior, and presentation independently testable.

## Canonical Models

The public models are:

- `FieldSchema`: name, data type, nullable, required, length, default, domain, alias, and source location.
- `DatasetSchema`: name, geometry type, record count, fields, subtype field, asset group field, asset type field, and source location.
- `DomainSpec` and `DomainValue`: domain name, code, description, and source location.
- `FieldMapping`: source field, target field, expression, lookup reference, default, and source location.
- `MappingPair`: mapping ID, source dataset, target dataset, definition query, purpose, expected count, field mappings, asset group/type, rationale, engineering context, and source location.
- `DataReferenceEntry`: source, target, mapping workbook, enabled state, attachment/global-ID settings, query, and source location.
- `AssetTypeSpec`: dataset, asset group/type names and codes, categories, terminals, and source location.
- `EngineeringContext`: network role, flow regime, material, lining, coating, dimensions, pressure class, slope, inverts, flow direction, installation method/date, lifecycle status, and extensible attributes.
- `EngineeringRule` and `EngineeringCondition`: rule ID, priority, conditions, target group/type, explanation, and review requirement.
- `AttributeRuleSpec`, `NetworkRuleSpec`, `TerminalSpec`, and `DirtyAreaRecord`.
- `Finding`: stable code, severity, validator, message, recommendation, dataset/field/mapping identifiers, source location, and structured details.
- `ValidationSummary` and `ValidationResult`: status, counts, executed validators, findings, input fingerprint, and toolkit version.

Model collections preserve logical order while reporters apply stable sorting. Input aliases are normalized case-insensitively, but original names and source locations remain available in findings.

## Project Configuration and Inputs

The primary entrypoint is a YAML project manifest:

```yaml
project:
  name: water_migration_readiness
  profile: water
inputs:
  source_schema: data/source_schema.csv
  target_schema: data/target_schema.xlsx
  mappings: data/source_target_mapping.xlsx
  source_domains: data/source_domains.csv
  target_domains: data/target_domains.csv
  asset_types: data/target_asset_types.csv
  data_reference: data/DataReference.xlsx
  attribute_rules: data/attribute_rules.csv
  network_rules: data/network_rules.csv
  terminals: data/terminals.csv
  dirty_areas: data/dirty_areas.csv
  engineering_rules: rules/engineering_rules.yml
validation:
  enabled:
    - schema
    - mapping
    - filters
    - domains
    - asset_classification
    - data_reference
    - attribute_rules
    - network_rules
    - dirty_areas
  fail_on: error
  severity_overrides: {}
outputs:
  directory: reports
  formats: [json, csv, markdown, html]
```

Only source schema, target schema, and mappings are required. Validators whose optional inputs are absent return an informational `SKIPPED_INPUT_MISSING` finding unless disabled explicitly. Paths resolve relative to the manifest directory.

CSV and workbook sheets use documented logical columns. Readers accept normalized aliases such as `SourceDefinitionQuery`, `source_definition_query`, and `definition_query`. XLSX auto-detection uses sheet-name and header scoring; ambiguous matches are configuration errors with candidate sheet names listed.

JSON and YAML inputs accept either row arrays or objects keyed by stable identifiers. Unknown columns are retained in an `extra` mapping for forward compatibility. Unknown top-level configuration keys are rejected to catch spelling errors.

## Validation Behavior

### Schema

Checks duplicate datasets and fields, supported data types, required metadata, nullability/default conflicts, geometry compatibility, subtype and asset field references, length compatibility, and target required fields.

### Mapping

Checks source and target dataset existence, source and target field existence, duplicate mappings, incompatible field types, unmapped required target fields, conflicting expressions/lookups, unsupported transformations, and mappings disabled by configuration.

### Definition Queries and Filters

The toolkit tokenizes a safe SQL-style subset to identify fields, operators, literals, parentheses, `AND`, `OR`, `NOT`, `IN`, `LIKE`, `IS NULL`, and comparison predicates. It checks referenced fields, balanced syntax, documented purpose, empty queries where required, expected counts, and overlap risk for simple equality/`IN` partitions. It does not execute queries or claim exhaustive semantic overlap detection.

### Domains

Checks domain existence, duplicate codes, code/description ambiguity, target-domain compatibility, explicit source-to-target value crosswalks, unmapped source values, invalid defaults, and asset group/type code membership.

### Asset Classification and Engineering Context

Checks that mapped asset groups/types exist in the target inventory and that a rationale is present when required. YAML engineering rules use `equals`, `not_equals`, `in`, `not_in`, `greater_than`, `greater_or_equal`, `less_than`, `less_or_equal`, `exists`, and `contains` operators. Rules are deterministic and ordered by priority. The validator reports no match, a single explained match, multiple conflicting matches, or a selected classification that differs from the rule result. Rules never silently overwrite a mapping.

### Data Reference

Checks required source/target/mapping references, duplicate enabled entries, missing files, mapping-pair consistency, query consistency, attachment/global-ID booleans, and disabled-row behavior.

### Attribute Rules

Checks target dataset and field dependencies, required mapped inputs, domain dependencies, triggering-event values, duplicate rule names, and readiness status. Arcade or Python expressions are stored and reported but not executed.

### Network Rules

Checks referenced datasets, asset groups/types, rule direction, terminal references, duplicate rules, self-contradictory entries, and coverage against mapped classifications. It validates declared metadata rather than building or tracing a network topology.

### Dirty Areas

Normalizes exported issue records, groups them by dataset/error code/severity, links known codes to configured remediation categories, detects missing identifiers, and produces actionable classifications. It does not open a Utility Network topology.

## Findings, Severity, and Status

Finding codes are stable uppercase identifiers grouped by validator, for example `MAP_TARGET_FIELD_UNKNOWN`, `FILTER_FIELD_UNKNOWN`, `DOMAIN_VALUE_UNMAPPED`, and `ASSET_RULE_CONFLICT`.

Severity levels are `info`, `warning`, and `error`. Project configuration may override severity by exact finding code. Findings are deduplicated using code plus affected identifiers and sorted by severity, validator, dataset, field, mapping ID, and source location.

Run status is:

- `pass`: no warnings or errors.
- `warning`: one or more warnings and no errors.
- `fail`: one or more errors.

CLI exit codes are `0` for pass, `1` for warning when `--fail-on-warning` is enabled, `2` for validation failure, and `3` for configuration/input/runtime errors.

## Public Interfaces

Python API:

```python
from pathlib import Path
from un_schema_qa import load_project, validate_project, write_reports

project = load_project(Path("project.yml"))
result = validate_project(project)
write_reports(result, formats=("json", "html"))
```

CLI:

```text
un-schema-qa validate PROJECT.yml [--output DIR] [--format FORMAT] [--fail-on-warning]
un-schema-qa inspect-workbook WORKBOOK.xlsx
un-schema-qa init DIRECTORY [--profile water|wastewater|stormwater]
un-schema-qa list-checks
un-schema-qa version
```

`validate` is the stable primary command. `inspect-workbook` reports detected sheets and column mappings without modifying the workbook. `init` creates a minimal runnable example project and refuses to overwrite existing files unless `--force` is supplied.

## Reporting

All report formats contain run status, toolkit version, project name, input fingerprint, validator list, finding counts, and findings.

- JSON is the canonical machine-readable output and follows a documented versioned schema.
- CSV contains one row per finding with JSON-encoded structured details.
- Markdown provides a concise summary and grouped finding tables.
- HTML is a self-contained, accessible report with embedded CSS, summary cards, validator sections, and no external scripts or network requests.

Report filenames are deterministic: `validation-report.json`, `.csv`, `.md`, and `.html`. Existing files are replaced only inside the configured output directory.

## Errors and Security

Configuration, input-format, workbook-detection, and validation-execution errors use typed exceptions with file, sheet, row, and column context. The CLI prints concise messages by default and tracebacks only with `--debug`.

The toolkit performs no network requests. It never evaluates arbitrary Python, Arcade, SQL, or template expressions from inputs. HTML values are escaped. Paths are resolved and validated before reading or writing. Generated examples contain synthetic infrastructure data only.

## Packaging and Quality

The distribution name is `utility-network-schema-qa-toolkit`, import package is `un_schema_qa`, and console command is `un-schema-qa`. Build backend is Hatchling. Runtime dependencies remain limited to Pydantic, Typer, PyYAML, openpyxl, and Jinja2. Development tooling uses pytest, pytest-cov, Ruff, and mypy.

Tests cover models, each reader, alias normalization, every validator, rule evaluation, engine status/severity behavior, each reporter, CLI exit codes, and end-to-end water/wastewater/stormwater example projects. CI runs lint, type checking, tests, coverage, build, and package metadata validation on supported Python versions.

V1.0 documentation includes README, changelog, contributing guide, code of conduct, security policy, architecture, configuration, input formats, validation rules, finding-code catalog, Python API, CLI, report schema, and three tutorials.

## Delivery Strategy

Implementation is split into independently reviewable milestones: repository foundation, models, readers, configuration, schema validation, mapping/filter validation, domain validation, engineering rules, remaining validators, engine/API, reporters, CLI, examples, documentation, CI, packaging, and release hardening. Each milestone receives a meaningful commit after its checks pass. Commit count is never padded with empty or cosmetic-only history.

V1.0 acceptance requires:

- Clean installation into a fresh virtual environment.
- Successful execution of all three example projects.
- Stable CLI and Python API behavior documented above.
- JSON, CSV, Markdown, and HTML output from the same validation result.
- Passing lint, type checks, tests, coverage threshold, package build, and metadata validation.
- No ArcGIS license or proprietary dependency.
- No placeholder modules, empty implementations, or README claims unsupported by code.
